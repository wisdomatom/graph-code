# 核心理念：为什么选择 GraphCode？

在数据爆炸与 AI 驱动的时代，传统的 REST/ORM 架构在处理**高度关联的数据**与**动态变化的业务逻辑**时，正面临着前所未有的挑战。

GraphCode 并非简单的底层图引擎封装层，而是一个**高度工程化的 GraphQL 中间层**，旨在解决复杂图谱应用中“开发效率、业务编排、AI 整合与安全治理”的四位一体难题。

---

## 1. 契约驱动：从“手工编码”到“Schema 驱动”

传统的 API 开发模式通常是：`数据库建模 -> 定义 DTO/POJO -> 编写 CRUD Controller -> 维护 API 文档`。每增加一个字段，都会引发全链路的修改。

**GraphCode 的哲学：Schema 即真相 (Single Source of Truth)。**
- **定义即交付**：你只需通过领域专用语言 (DSL) 定义业务模型，系统会自动生成高性能、类型安全的 GraphQL 查询与修改接口。
- **关联引擎**：原生支持无限层级的嵌套查询与原子化关联写入。无论是 `_CREATE`、`_CONNECT` 还是 `_DISCONNECT`，复杂的图关系维护由翻译层自动处理，开发者无需编写一行底层指令。

---

## 2. GraphRAG 时代的 AI 原生支持

随着大语言模型 (LLM) 的普及，纯向量检索（Vector Search）的局限性日益显现：它缺乏对领域知识结构的理解。

**GraphCode 深度整合了向量检索与关系图谱：**
- **原生向量类型**：在 Schema 中直接定义 `@type(float32vector)`，支持点积、余弦等多种度量。
- **加权重排 (Vector Math)**：支持在图检索的基础上进行自定义评分重排（如：`title_SCORE * 0.8 + content_SCORE * 0.2`），实现真正的 **GraphRAG**。
- **变量隔离翻译**：自动处理嵌套查询中的变量作用域与底层语法陷阱，确保在高并发查询下 AI 召回的精准度。

---

## 3. BT 业务编排：解耦“硬编码”逻辑

当业务逻辑变得复杂时（如涉及多步校验、外部调用、事务回滚），传统的 Go/Java 代码会迅速退化为不可维护的“面条代码”。

**GraphCode 引入了行为树 (Behavior Tree) 编排引擎：**
- **逻辑流转可视化**：将业务逻辑从代码逻辑中剥离，通过配置文件实现节点流转。
- **数据流 (Blackboard) 模式**：利用 `jq` 进行节点间的数据清洗与转换，实现真正的低代码业务流。
- **原子化事务**：支持请求级事务共享，确保 BT 中的多个 GraphQL 操作在一个原子事务内提交或回滚。

---

## 4. 工业级安全与审计 (ACL & Audit)

在企业级应用中，权限与审计往往是最后被考虑、却又最难实现的环节。

**GraphCode 将治理能力内建于内核：**
- **侧信道防护**：通过权限预计算与内部变量注入，彻底杜绝用户通过排序顺序或统计信息推测无权访问数据的可能性（推理攻击）。
- **行级隔离 ($user)**：支持基于登录用户身份的动态过滤器，在数据库层面实现物理隔离，而非在应用层进行结果过滤。
- **增量审计**：审计记录采用“Delta Only”技术，仅存储变更前后的差异，既节省存储空间，又为故障溯源提供了极高的可读性。

---

GraphCode 的目标是让开发者从繁杂的底层数据映射与逻辑流转中解脱出来，专注于业务价值的创造。如果你正在构建**智能知识图谱、复杂的商城系统、或者是需要高性能权限隔离的 SaaS 平台**，GraphCode 将是你的不二之选。
