# BT Business Orchestration (Behavior Tree)

In complex business systems, simple CRUD interfaces often fall short of supporting intricate business logic. GraphCode introduces the **BT (Behavior Tree) Business Orchestration Engine**, allowing developers to declaratively combine instructions via YAML to achieve ultimate decoupling between data operations and business processes.

---

## 1. Core Definition Components

A Behavior Tree is described logically through a YAML file, with its core fields as follows:

### 1.1 Basic Metadata
*   **`name`**: The name of the behavior tree. it serves as the prefix for the generated business interface name.
*   **`version`**: The business logic version number. Since behavior trees are essentially business logic, they support version iteration.
    *   **Instantiation Mechanism**: When a behavior tree executes, it first **instantiates** a tree based on the current YAML template. Once instantiated, that specific execution strictly follows the snapshot logic at that time, unaffected by subsequent version upgrades, ensuring execution consistency.

### 1.2 Blackboard Definition (`blackboard`)
The blackboard defines the **rules** for data transfer between nodes (based on `jq` expressions).
*   **`jq_input`**: Defines the **input construction** before each node runs. It extracts data from the global blackboard or user info and transforms it into variables required for that node's execution.
*   **`jq_output`**: Defines the **result processing** after each node runs.
    *   **Local Promotion**: Used to extract local data generated by a sub-node's execution into the **Global Blackboard** for reference by subsequent nodes.
    *   **Result Customization**: **A Key Feature**. Behavior tree generated interfaces do not have a fixed return format. By defining the `jq_output` of the root node (typically with ID `"00001"`), developers can precisely customize the final data structure returned to the frontend.

### 1.3 Root Node Definition (`root`)
*   **`funcIO`**: The input/output parameter Schema definition for the behavior tree. The syntax is largely consistent with [Schema DSL](./schema-dsl.md), but since these are memory objects, **database directives like `@index` are not required**.
*   **`children`**: Defines a list of sub-nodes. By recursively nesting the `children` parameter, any shape of business logic tree can be constructed.
*   **`doFunc`**: The business core of node execution. Supports **`graphql`**, **`http`**, and **`grpc`** types, covering the vast majority of microservice orchestration scenarios.

---

## 2. Runtime Context

During the execution of a behavior tree, the system maintains two levels of state:

*   **Global Blackboard (`blackboard`)**: Shared throughout the entire lifecycle. Contains the user's original input (`__input`), authentication info (`__auth`), and custom variables written back by nodes.
*   **Local Blackboard (`localBlackboard`)**: Valid only for the current node. The `__output` field stores the raw result returned by the current node's executor.

## 3. Full Lifecycle Flow: Example "Create Order"

### 3.1 Step 1: Define Behavior Tree (`upsertMetaBT`)
Submit the YAML definition via the `upsertMetaBT` interface.

```graphql
mutation{
  upsertMetaBT(input:"""
name: "CreateOrder"
version: "v20250711"
blackboard:
  jq_input:
    "00001-00002": >
      .blackboard.__auth.user.id as $userID |
      {
        userID:$userID
      }
    "00001-00003": >
      .blackboard.__input.input as $input |
      {
        input:{
          uuid: .blackboard.__uuid,
          createdAt: .blackboard.__createdAt,
          status: "unpaid",
          description: $input.remark,
          address_CONNECT: {
            where: {
              id: $input.addressId
            }
          },
          items_CONNECT: {
            where: {
              id: .blackboard.__items_id
            }
          },
          user_CONNECT: {
            where: {id: .blackboard.__auth.user.id}
          }
        }
      }
    "00001-00004": >
      {
        id: .blackboard.__items_id
      }
  jq_output:
    "00001": >
      {
        uuid:.blackboard.__uuid,
        createdAt:.blackboard.__createdAt
      }
    "00001-00001": >
      {
        __uuid:.localBlackboard.__output.systemInfo.uuid,
        __createdAt:.localBlackboard.__output.systemInfo.time_rfc3339
      }
    "00001-00002": >
      {__items_id:.localBlackboard.__output.queryCarts[0].items[].id}
    "00001-00003": >
      {createOrders:.localBlackboard.__output}
root:
  deprecated: false
  tickType: ""
  funcIO: |
    Input{
      addressId @type(string) @required
      paymentMethod @type(string) @required
      remark @type(string) @required
    }
    Output{
      input @type(string) @isScalar
      output @type(string) @isScalar
    }
  children:
    - doFunc:
        type: "graphql"
        graphql: |
          {
            systemInfo{
              uuid
              time_rfc3339
            }
          }
    - doFunc:
        type: "graphql"
        graphql: |
          query($userID:String!){
            queryCarts(where:{
              userID:$userID
            }){
              userID
            	items(where:{selected:true}){
                id
                quantity
                product{
                  id
                  name
                  price
                  stock
                }
              }
            }
          }
    - precheck: ""
      doFunc:
        type: "graphql"
        graphql: |
          mutation($input:OrderCreate!){
            createOrders(input:[$input]){
              nodeCreated
              orders{
                id
              }
            }
          }
    - precheck: ""
      doFunc:
        type: "graphql"
        graphql: |
          mutation($id:String!){
            updateItems(
              where:{
                id:[$id]
              }
              update:{
                cart_DISCONNECT:{}
              }
            ){
              nodeUpdated
            }
          }
    """)
}
```

### 3.2 Step 2: Publish Interface (`generateAndPublishMetaBTFunc`)
After submitting the definition, call the publish interface to generate the corresponding GraphQL function in real-time.

```graphql
mutation {
  generateAndPublishMetaBTFunc
}
```

### 3.3 Step 3: Use Business Interface
Once published, the system automatically generates a new API following the `Name__Version` naming convention. The `data` content it returns is entirely defined by the `jq_output` of the root node (`00001`):

```graphql
mutation {
  # Format: Name__VersionNumber
  CreateOrder__v20250711(
    input: {
      addressId: "0x2a"
      remark: "Please deliver in the afternoon, pay attention to product color"
    }
  )
}
```

## 4. Best Practices

1.  **Minimum Input Principle**: Only have the user pass mandatory parameters. All logic determinations (e.g., initial status, internal associations) should be completed within the BT.
2.  **Security Isolation**: For write operations involving core assets (e.g., balance, orders), it is recommended to disable direct Mutation permissions at the model level and only allow access through published BT interfaces.
3.  **Node Decoupling**: Try to have each node perform only one operation, with all complex logic transformations handled within `jq_input`.
